Back-end (Rails 8 API-only)

Versioned namespace, every endpoint documented by an rswag request spec that emits swagger/v1/swagger.yaml as a side effect. Auth is a stateless JWT — login (password or Google) returns a Bearer token, the SPA sends it in the Authorization header on every write. No cookies, no CSRF middleware: the header is not auto-attached cross-origin, so CSRF is structurally out of reach.

Authentication & Authorization

Stateless JWTs, no cookies, two login surfaces (password + Google), and a role split so non-admin users can sign up without unlocking the CMS.

Auth-aware scope on the API base controller

Public callers see only published records. The same URL returns drafts to an authenticated admin. One method on BaseController, every content controller reuses it. 

# app/controllers/api/v1/base_controller.rb
def visible_scope_for(model_class)
  current_user&.admin? ? model_class.order(:position) : model_class.published
end

# usage
def index
  render_success(projects: ProjectBlueprint.render_as_hash(visible_scope_for(Project)))
end

The JWT auth interceptor

One interceptor wires Authorization: Bearer ... onto every outbound request when a token is present in TokenService. No cookies, no withCredentials, no CSRF surface — the header is not auto-attached cross-origin, so attackers can't ride it. 

// core/interceptors/auth.interceptor.ts
export const authInterceptor: HttpInterceptorFn = (req, next) => {{ '{' }}
  const token = inject(TokenService).get();
  return token
    ? next(req.clone({{ '{' }} setHeaders: {{ '{' }} Authorization: `Bearer ${{ '$' }}{{ '{' }}token{{ '}' }}` {{ '}' }} {{ '}' }}))
    : next(req);
{{ '}' }};

Stateless JWT — one service, two methods

Login signs a JWT with secret_key_base; every subsequent request decodes the Authorization: Bearer header in the Authentication concern. No sessions, no cookies, no SameSite/Secure dance to keep dev and prod in sync. 

# app/services/json_web_token.rb
class JsonWebToken
  ALGORITHM = "HS256"
  DEFAULT_EXP = 7.days

  def self.encode(payload, exp: DEFAULT_EXP.from_now)
    JWT.encode(payload.merge(exp: exp.to_i), secret, ALGORITHM)
  end

  def self.decode(token)
    JWT.decode(token, secret, true, algorithm: ALGORITHM).first
  rescue JWT::DecodeError, JWT::ExpiredSignature
    nil
  end

  def self.secret = Rails.application.secret_key_base
end

authGuard & adminGuard — return a UrlTree, not a navigate()

Functional guards. Returning a UrlTree lets Angular handle the redirect itself — no race conditions with the route's own navigation and no imperative router.navigate() hiding inside a guard. Two guards: authGuard protects anything that requires a logged-in user; adminGuard is stricter and bounces non-admin visitors to the home page instead of leaving them in a permission limbo. 

// core/guards/auth.guard.ts
export const authGuard: CanActivateFn = () => {{ '{' }}
  const auth = inject(AuthService);
  const router = inject(Router);
  return auth.isAuthenticated() ? true : router.createUrlTree(['/admin/login']);
{{ '}' }};

// core/guards/admin.guard.ts
export const adminGuard: CanActivateFn = () => {{ '{' }}
  const auth = inject(AuthService);
  const router = inject(Router);
  if (!auth.isAuthenticated()) return router.createUrlTree(['/admin/login']);
  if (!auth.isAdmin())         return router.createUrlTree(['/']);
  return true;
{{ '}' }};

Login hardening — three layers, no security through obscurity

The admin route is public; the defenses are not. Rate limiting via rack-attack caps brute-force attempts at 5 per IP per 5 minutes. Per-account lockout survives IP rotation: failed_attempts and locked_until on the user track the counter, and 15 minutes of lockout kicks in after 5 misses. Timing-safe auth via Rails 7.1+ User.authenticate_by runs bcrypt whether or not the email exists, so response time can't enumerate accounts. The locked response includes locked_until so the frontend can show a real countdown. 

# config/initializers/rack_attack.rb
class Rack::Attack
  throttle("login/ip", limit: 5, period: 5.minutes) do |req|
    req.ip if req.post? && req.path == "/api/v1/session"
  end
end

# app/models/user.rb
MAX_FAILED_ATTEMPTS = 5
LOCKOUT_DURATION    = 15.minutes

def locked? = locked_until.present? && locked_until > Time.current

def register_failed_attempt!
  new_count = failed_attempts + 1
  if new_count >= MAX_FAILED_ATTEMPTS
    update_columns(failed_attempts: new_count, locked_until: LOCKOUT_DURATION.from_now)
  else
    update_columns(failed_attempts: new_count)
  end
end

# app/controllers/api/v1/sessions_controller.rb
def create
  existing = User.find_by(email: params[:email])
  if existing&.locked?
    return render_error(
      "Account temporarily locked.",
      status: :unauthorized,
      extra: {{ '{' }} locked_until: existing.locked_until.iso8601 {{ '}' }}
    )
  end

  # bcrypt runs whether or not the user exists → no timing enumeration
  user = User.authenticate_by(email: params[:email], password: params[:password])
  if user
    user.reset_failed_attempts!
    render_success(user: UserBlueprint.render_as_hash(user), token: issue_token(user))
  else
    existing&.register_failed_attempt!
    render_error("Invalid credentials", status: :unauthorized)
  end
end

Sign in with Google — verify server-side, never trust the client

Frontend uses Google Identity Services to obtain a signed ID token client-side. The backend re-verifies it against Google's public keys (the token is JWS-signed; the SDK return value alone proves nothing without re-verification). On verify, the controller finds the user by google_uid, links Google to an existing email-only admin if there's a match, or creates a brand-new public user with role: "user". Admin promotion is never automatic. 

# app/services/google_id_token_verifier.rb
class GoogleIdTokenVerifier
  include ServiceResult

  def call(id_token)
    return error("Missing id_token") if id_token.blank?
    client_id = Rails.application.credentials.dig(:google, :client_id)
    payload = GoogleIDToken::Validator.new.check(id_token, client_id)
    success(payload)
  rescue GoogleIDToken::ValidationError => e
    error("Invalid Google token: #{{ '{' }}e.message{{ '}' }}")
  end
end

# app/controllers/api/v1/sessions_controller.rb
def create_google
  result = GoogleIdTokenVerifier.new.call(params[:id_token])
  return render_error("Invalid Google token", status: :unauthorized) unless result[:ok]

  payload = result[:data]
  return render_error("Google email is not verified", status: :unauthorized) \
    unless payload["email_verified"]

  user = find_or_create_google_user(payload)
  user.reset_failed_attempts!
  render_success(user: UserBlueprint.render_as_hash(user), token: issue_token(user))
end

API Patterns

Two response helpers, every endpoint, one error shape. Tests double as OpenAPI docs. Allowlists are tight and explicit.

Two helpers, every endpoint — render_success + render_error

Every controller funnels through exactly two helpers on BaseController. Payloads handed to render_success are always Blueprinter hashes — the helper owns the envelope (status + json wrapper), never serialization. render_error accepts an AR record, a string, or an array, and always emits {{ '{' }} errors: [...] {{ '}' }} so the frontend can type-narrow on a single shape. Pass extra: for structured side-info like locked_until. 

# app/controllers/api/v1/base_controller.rb
def render_success(status: :ok, **payload)
  render json: payload, status: status
end

def render_error(error, status: :unprocessable_entity, extra: {{ '{' }}{{ '}' }})
  errors =
    case error
    when ActiveRecord::Base then error.errors.as_json(full_messages: true)
    when Array then error.map(&:to_s)
    else [ error.to_s ]
    end
  render json: {{ '{' }} errors: errors, **extra {{ '}' }}, status: status
end

# app/controllers/api/v1/projects_controller.rb
def create
  project = Project.new(project_params)
  if project.save
    render_success(project: ProjectBlueprint.render_as_hash(project), status: :created)
  else
    render_error(project)            # 422 with AR errors
  end
end

rswag — the request spec IS the API doc

Every endpoint has a request spec in the rswag DSL. The same file runs assertions and emits swagger/v1/swagger.yaml. "Document the API later" is no longer a thing on this codebase. 

# spec/requests/api/v1/projects_spec.rb
path "/api/v1/projects" do
  get("List projects (published only when public; all when authenticated)") do
    tags "Projects"
    produces "application/json"

    response(200, "ok") do
      schema type: :object,
             properties: {{ '{' }} projects: {{ '{' }} type: :array, items: PROJECT_SCHEMA {{ '}' }} {{ '}' }},
             required: %w[projects]

      before do
        create(:project, title: "Public", published: true)
        create(:project, title: "Hidden Draft", published: false)
      end

      run_test! do |response|
        titles = JSON.parse(response.body)["projects"].map {{ '{' }} |p| p["title"] {{ '}' }}
        expect(titles).to include("Public")
        expect(titles).not_to include("Hidden Draft")
      end
    end
  end
end

CORS — narrow allowlist, no credentials needed

No CSRF middleware. With JWT in the Authorization header (not a cookie), browsers don't auto-attach it cross-origin — CSRF is structurally impossible. The CORS allowlist exists to limit who can even reach the API. 

# config/initializers/cors.rb
Rails.application.config.middleware.insert_before 0, Rack::Cors do
  allow do
    origins "http://localhost:4200", "http://127.0.0.1:4200",
            "https://zachyoung.com", "https://www.zachyoung.com"

    resource "*",
      headers: :any,
      expose:  %w[Authorization],
      methods: %i[get post put patch delete options head]
  end
end

Service Architecture

Service objects return a normalized envelope, so callers pattern-match on :ok instead of branching on exceptions, nils, and sentinels.

ServiceResult — one envelope every service returns

Each service exposes a single #call. The result is always either {{ '{' }} ok: true, data: ... {{ '}' }} or {{ '{' }} ok: false, error: ... {{ '}' }}. The module is callable as ServiceResult.success(...) directly or as bare success(...) / error(...) once included. 

# app/services/service_result.rb
module ServiceResult
  module_function

  def success(data = nil) = {{ '{' }} ok: true,  data: data {{ '}' }}
  def error(error)        = {{ '{' }} ok: false, error: error {{ '}' }}
end

# Example service
class PublishPostService
  include ServiceResult

  def call(post)
    return error("Post not found") if post.nil?
    post.update!(published: true)
    success(post: post)
  rescue ActiveRecord::RecordInvalid => e
    error(e.record)                 # render_error speaks AR records natively
  end
end

# Controller — pattern-match on :ok, hand the payload to render_*
result = PublishPostService.new.call(post)
result[:ok] \
  ? render_success(post: PostBlueprint.render_as_hash(result[:data][:post]))
  : render_error(result[:error])

Frontend Patterns

Signals for state, afterNextRender for browser-only setup, CSS custom properties for theming so we don't need a single style recalc in JS to swap modes.

ThemeService — one effect syncs three things

signal() for state, effect() for side-effects. Every theme change updates my own custom-property class, Bootstrap's data-bs-theme attribute, and localStorage in one pass. SSR-safe via isPlatformBrowser. 

constructor() {{ '{' }}
  effect(() => {{ '{' }}
    const t = this._theme();
    const root = this.doc.documentElement;
    root.classList.remove('theme-light', 'theme-dark');
    root.classList.add(`theme-${{ '$' }}{{ '{' }}t{{ '}' }}`);
    root.setAttribute('data-bs-theme', t);
    if (isPlatformBrowser(this.platformId)) {{ '{' }}
      localStorage.setItem('theme', t);
    {{ '}' }}
  {{ '}' }});
{{ '}' }}

Theme tokens — CSS custom properties, not Sass variables

Sass variables resolve at build time and don't react to a class change. CSS custom properties resolve at runtime. Switching one class on <html> recolors every component without a single runtime style recalc in JS. 

// styles.scss
:root, .theme-light {{ '{' }}
  --bg:          #ffffff;
  --bg-surface:  #e9ecef;
  --fg:          #111111;
  --accent:      #4b9cd3;
{{ '}' }}

.theme-dark {{ '{' }}
  --bg:          #18181b;
  --bg-surface:  #27272a;
  --fg:          #fafafa;
  --accent:      #4b9cd3;
{{ '}' }}

// any component
.card {{ '{' }} background: var(--bg-surface); color: var(--fg); {{ '}' }}

ParallaxDirective — afterNextRender, signal inputs, reduced-motion

Browser-only setup without an isPlatformBrowser guard: afterNextRender only runs in the browser, so the scroll listener never registers during SSR. Signal input() with an alias keeps the public attribute terse, rAF throttling caps work at one update per frame, and the whole effect opts out under prefers-reduced-motion: reduce. 

// shared/directives/parallax.directive.ts
@Directive({{ '{' }} selector: '[appParallax]', standalone: true {{ '}' }})
export class ParallaxDirective {{ '{' }}
  readonly factor = input(0.3, {{ '{' }} alias: 'appParallax' {{ '}' }});

  constructor() {{ '{' }}
    afterNextRender(() => {{ '{' }}
      if (window.matchMedia('(prefers-reduced-motion: reduce)').matches) return;
      let ticking = false;
      const update = () => {{ '{' }}
        // translate3d clamped to ±factor * viewportH (no overflow past section)
        ticking = false;
      {{ '}' }};
      window.addEventListener('scroll', () => {{ '{' }}
        if (!ticking) {{ '{' }} ticking = true; requestAnimationFrame(update); {{ '}' }}
      {{ '}' }}, {{ '{' }} passive: true {{ '}' }});
    {{ '}' }});
  {{ '}' }}
{{ '}' }}

Domain Models

AR models lean on Rails built-ins (normalizes, validations) instead of bespoke callbacks. Frontend models graduate from interfaces to real classes when they need to carry behavior.

ActiveRecord normalizes: (Rails 7.1+)

Built-in attribute normalization. No before_validation callback, no custom setter, no "is the email lower-cased on this read or that one" inconsistency. 

# app/models/contact_message.rb
class ContactMessage < ApplicationRecord
  normalizes :name, :message, with: ->(v) {{ '{' }} v.strip {{ '}' }}
  normalizes :email,           with: ->(v) {{ '{' }} v.strip.downcase {{ '}' }}

  validates :name, :message, presence: true
  validates :email, presence: true,
                    format: {{ '{' }} with: URI::MailTo::EMAIL_REGEXP {{ '}' }}
end

Project — from interface to class with a time_ago field

The frontend's Project graduated from a structural interface to a real class so it can carry behavior. ProjectData still describes the raw API shape; the class wraps it, exposes user-set project_start / project_end dates from the backend, and computes time_ago once in the constructor against project_end. Everything is readonly — the value is deterministic at construction, so a stored field is cheaper than a getter and reads the same in templates: {{ '{' }}{{ '{' }} project.time_ago {{ '}' }}{{ '}' }}. The ProjectsService maps every API response through new Project(p) so consumers always receive real instances. 

// core/models/project.model.ts
export interface ProjectData {{ '{' }}
  id: number;
  title: string;
  // …
  project_start: string | null;   // 'YYYY-MM-DD' from the API
  project_end:   string | null;
{{ '}' }}

export class Project {{ '{' }}
  readonly id: number;
  readonly title: string;
  // …
  readonly project_start: string | null;
  readonly project_end: string | null;
  readonly time_ago: string;

  constructor(data: ProjectData, now: Date = new Date()) {{ '{' }}
    this.id = data.id;
    this.title = data.title;
    // …copy the rest…
    this.project_start = data.project_start;
    this.project_end   = data.project_end;
    this.time_ago = Project.computeTimeAgo(now, data.project_end);
  {{ '}' }}

  private static computeTimeAgo(now: Date, project_end: string | null): string {{ '{' }}
    if (!project_end) return '';
    const days = Math.floor((now.getTime() - new Date(project_end).getTime()) / 86_400_000);
    if (days < 1) return 'today';
    if (days < 7) return days + ' day' + (days === 1 ? '' : 's') + ' ago';
    // …walks days → weeks → months → years and picks the right unit
  {{ '}' }}
{{ '}' }}
